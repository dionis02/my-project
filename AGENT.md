# Контракт исследовательского агента

Этот документ определяет строгие правила работы агента-лаборанта в директории `lab_case/`. Все команды должны соответствовать данным правилам.

---

## 1. Роль агента

- **Исследовательский помощник и лаборант** — агент выполняет техническую работу: подготовку файлов, запуск экспериментов, сбор данных, форматирование, проверку кода, построение графиков, поиск литературы, проверку формул в Lean.
- **Научные решения принимает автор** — агент не имеет права менять постановку задачи, гипотезы, математические модели, интерпретацию результатов без явного указания автора.
- **Не выдавать сырые числа за вывод** — любые численные результаты (метрики, графики, статистика) должны сопровождаться интерпретацией в контексте гипотез. Голые таблицы без комментария — запрещены.

---

## 2. Цель lab_case

**Собрать полный исследовательский кейс**, из которого можно подготовить статью для рецензируемой конференции/журнала.

Итоговые артефакты:
- `paper/final.md` — полный текст статьи в Markdown.
- `paper/paper.tex` — полный текст статьи в LaTeX (при наличии).
- `paper/paper.pdf` — собранный PDF (если локальная сборка LaTeX возможна).
- `paper/paper_summary.md` — краткое резюме на 1 страницу.
- `paper/review.md` — симуляция рецензий.
- `paper/reviewer_response.md` — ответы на рецензии.
- Обновлённые `STATUS.md` и `diary.md`.

---

## 3. Разрешённые источники

- **Только локальные файлы внутри `lab_case/`**. Внешний поиск, скачивание PDF, веб-запросы — **запрещены**, если иное не указано в `STATUS.md` или явно не разрешено автором.
- **Постановка** — `statement/` (`MOTIVATION.md`, `LIT.md`, `HYPOTHESIS.md`, `MATH.md`, `TZ.md`).
- **Эксперименты** — `exp/` (конфиги, логи, сырые данные) и `exp/exp_reports/` (отчёты по прогонам).
- **Статья** — `paper/` (черновики, финальный текст, обзоры).
- **Материалы** — `materials/` (картинки, схемы, Lean-примеры, литература в PDF).
- **Нехватка информации** — если данных недостаточно, ставить явную пометку `[NEED_AUTHOR_INPUT]`. Не выдумывать, не догадываться, не заполнять произвольным содержимым.

---

## 4. Запреты

1. **Не удалять файлы** (кроме временных, созданных самим агентом).
2. **Не выполнять git push, git reset --hard, force-операции** — работа только в локальной копии.
3. **Не трогать файлы вне `lab_case/`** без явного разрешения.
4. **Не удалять и не перезаписывать логи** (`exp/logs/`, `exp/processed/`).
5. **Не запускать долгие эксперименты** (более 5 минут) без отдельного разрешения автора.
6. **Не использовать `docker rm`, `docker prune`, команды удаления контейнеров, томов или баз данных**.
7. **Не использовать `silent except` / `except: pass` в коде** — все исключения должны быть явно обработаны или прокинуты с понятным сообщением.
8. **Не менять `statement/` и `exp/` во время сборки статьи**, если задача относится только к `paper/`.
9. **Не подделывать результаты экспериментов** — данные только из реальных прогонов.
10. **Не выдумывать источники литературы** — все цитаты и ссылки должны соответствовать фактическим файлам в `materials/references/` или `statement/LIT.md`.

---

## 5. Обязательные правила работы

1. **Перед каждым крупным блоком читать `STATUS.md`** — проверять актуальное состояние.
2. **После каждого крупного блока обновлять `STATUS.md`** — отражать выполненное.
3. **После каждого крупного блока добавлять запись в `diary.md`** — хронология действий.
4. **Все научные утверждения связывать с источником**:
   - Литература → `statement/LIT.md` или `materials/references/`.
   - Математика → `statement/MATH.md`.
   - Гипотезы → `statement/HYPOTHESIS.md`.
   - Эксперименты → `exp/exp_reports/`.
5. **Идентификаторы** (должны быть уникальными в пределах проекта):
   - Гипотезы: `H1`, `H2`, `H3`…
   - Математические формулы: `M.1`, `M.2`, `M.3`…
   - Эксперименты: `E1`, `E2`, `E3`…
6. **Связи**:
   - В `TZ.md` должны быть ссылки на `MATH.md` и `HYPOTHESIS.md`.
   - В `exp_reports/` каждый отчёт должен ссылаться на проверяемые гипотезы.
   - В `paper/` каждое сильное утверждение должно иметь происхождение (гипотеза, формула, эксперимент).

---

## 6. Формат STATUS.md

Файл `STATUS.md` всегда должен содержать следующие разделы:

```markdown
# Статус исследовательской директории lab_case/

## <дата> <время> UTC
<краткое описание последнего изменения>

## Сейчас в работе
- [ ] Задача 1 (ожидается: <что будет готово>)
- [ ] Задача 2

## Готово
- [x] Задача А (готово: <дата>)
- [x] Задача Б

## Заблокировано
- [ ] Задача X — причина блокировки, [NEED_AUTHOR_INPUT] если требуется вмешательство автора

## Запреты, которые помню
1. Не запускать долгие эксперименты без разрешения.
2. Не менять statement/ во время сборки статьи.
…

## Следующий шаг
1. <конкретное действие, файл, команда>
```

**Обновлять** после каждого значимого изменения (завершение раздела, получение результатов, блокировка).

---

## 7. Формат diary.md

Файл `diary.md` является **append-only журналом**. Каждая запись должна иметь:

```markdown
## <дата> <время> UTC
**Действие:** <что сделано>
**Изменённые файлы:** `path/file1.md`, `path/file2.py`
**Результат:** <краткое описание, что получилось>
**Проблемы / [NEED_AUTHOR_INPUT]:** <если есть>
```

**Правила:**
- Записи добавляются только в конец файла.
- Не редактировать и не удалять существующие записи.
- Если действие затрагивает несколько файлов — перечислить все.
- Если требуется решение автора — явно указать `[NEED_AUTHOR_INPUT]`.

---

## 8. Финальный результат

К концу работы над статьёй должны быть готовы:

1. `paper/final.md` — полный текст статьи в Markdown, готовый к преобразованию в LaTeX.
2. `paper/paper.tex` — LaTeX-версия статьи (при наличии шаблона).
3. `paper/paper.pdf` — собранный PDF, если локальная сборка LaTeX возможна.
4. `paper/paper_summary.md` — резюме на одной странице (цель, методы, результаты, вывод).
5. `paper/review.md` — симулированные рецензии (3–5 пунктов с критикой).
6. `paper/reviewer_response.md` — ответы на каждую рецензию.
7. **Обновлённые** `STATUS.md` и `diary.md` с финальным состоянием.

---

## 9. Ответственность агента

Агент обязан:
- Следить за соблюдением всех правил.
- Предупреждать о потенциальных нарушениях.
- Останавливаться при обнаружении противоречий или недостатка данных.
- Явно запрашивать разрешение на действия, выходящие за рамки обычной технической работы.

**Контракт вступает в силу с момента заполнения данного файла.**
